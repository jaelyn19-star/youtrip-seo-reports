Fix Homepage hero image — LCP 25.3s mobile (10× over threshold)
Homepage · Mobile LCP 25.3s · Perf score 36 Core Web Vitals Dev: ~2–4 hours 📅 Apr 1 — Dev Ticket → See CWV report

The homepage is the worst performer in the audit at 25.3s mobile LCP — nearly 10× over Google's "poor" threshold of 4s. The 8× gap between mobile (25.3s) and desktop (3.0s) on the same page confirms this is a mobile-specific asset problem, not server latency. Almost certainly caused by a large, unoptimised hero image loading without priority hints.

  • Step 1 — Identify the LCP element: Open Chrome DevTools → Performance tab → set throttling to "Slow 3G" and CPU 4× slowdown → record a page load → click the LCP marker in the timeline to see which element is the LCP candidate (usually the hero banner or above-the-fold image)
  • Step 2 — Check image weight: In the Network tab, filter by "Img" — the hero image should be under 100KB for mobile. If it's a PNG or JPG over 300KB, compress and convert to WebP via Squoosh (squoosh.app) before re-uploading
  • Step 3 — Add fetchpriority: On the <img> tag for the hero, add fetchpriority="high" — this tells the browser to request this image before anything else in its queue
  • Step 4 — Add preload in <head>: <link rel="preload" as="image" href="/hero-mobile.webp" fetchpriority="high"> — belt-and-suspenders fix that starts the download before the browser parses the body
  • Note: If the hero is a CSS background-image, replace it with an <img> tag — CSS backgrounds cannot receive fetchpriority and the browser discovers them late
  • Milestone targets: LCP under 4s (exit "Poor" zone) as first win; under 2.5s (Google "Good") as the final target

Defer non-critical JS on About YouTrip — TBT 1,120ms blocking mobile
About YouTrip · Mobile TBT 1,120ms · Perf score 41 Core Web Vitals Dev: ~2–3 hours 📅 Apr 1 — Dev Ticket → See CWV report

Total Blocking Time of 1,120ms on the About YouTrip page means the main thread is locked for over a second while loading — the page is completely unresponsive to user input during this window. TBT is Google's lab proxy for INP (Interaction to Next Paint), which became a Core Web Vital in 2024. The likely culprits are third-party analytics and marketing scripts loading synchronously.

  • Step 1 — Profile the page: Chrome DevTools → Performance tab → throttle to "Mobile (Slow 4G)" → record load → inspect the "Main" thread — look for red triangles labeled "Long Task" (any JS execution over 50ms). Screenshot or note the script names causing each long task
  • Step 2 — Audit GTM container: Open Google Tag Manager for this property → check which tags fire on "All Pages" or "Page View" — common offenders are Hotjar, Meta Pixel, LinkedIn Insight Tag, chat widgets, and ad network scripts. None of these need to block page rendering
  • Step 3 — Change GTM trigger timing: Move non-critical tags from "Page View" to "DOM Ready" or "Window Loaded" — this defers them until after the page is interactive without removing the tracking
  • Step 4 — Add defer to inline scripts: Any <script src="..."> tags outside GTM should have defer added: <script src="analytics.js" defer>
  • Step 5 — WP Rocket (if installed): Enable "Delay JavaScript Execution" → exclude only critical scripts (e.g. jQuery if needed for interactivity) and defer everything else
  • Target: TBT under 300ms on mobile (from 1,120ms). Even getting to 600ms would move the perf score from 41 to ~60+

Verify mobile caching is active — TTFB contributing to LCP on all pages
Sitewide · Mobile TTFB issue Core Web Vitals / Server ~30–60 minutes → See CWV report

Elevated LCP across all pages suggests the server is not delivering cached HTML to mobile user-agents — meaning every mobile load starts with a full PHP and database execution cycle before a single byte of HTML is sent. This is a lower-ceiling fix: it won't fully solve LCP on its own, but it removes a compounding factor that inflates every other metric.

  • Step 1 — Check caching plugin settings: In WP Rocket (or LiteSpeed Cache / W3 Total Cache), go to the "Mobile" settings tab and confirm "Enable caching for mobile devices" is turned on. Some plugins create a separate mobile cache; others serve the same cached file — verify yours is set up correctly
  • Step 2 — Test with curl: Run curl -A "Mozilla/5.0 (iPhone; CPU iPhone OS 15_0)" -I https://www.you.co/sg/ and check the response headers for X-Cache: HIT or CF-Cache-Status: HIT (Cloudflare). A MISS means caching isn't working for mobile
  • Step 3 — CDN / Cloudflare check: If using Cloudflare, confirm "Cache Level" is set to "Standard" or "Cache Everything" and that mobile user-agents aren't being bypassed by a Page Rule
  • Step 4 — Purge and warm cache: After any config changes, purge the cache and load 5–6 pages manually to warm it, then re-test with curl to confirm HITs

Audit and defer Homepage third-party scripts — desktop TBT 2,340ms
Homepage desktop · TBT 2,340ms · Perf score 43 Core Web Vitals Dev: ~2–3 hours → See CWV report

A desktop TBT of 2,340ms is severe — 11× over Google's 200ms threshold. Unlike the mobile issues which are partly image-driven, this desktop score is almost entirely caused by synchronous JavaScript execution. The homepage desktop perf score of 43 means even fast-connection users experience an unresponsive page during load. This will impact INP, which is now a Core Web Vital in Google's ranking signal.

  • Step 1 — Use PageSpeed Insights Opportunities: Run PSI on the homepage (desktop) → open the "Opportunities" and "Diagnostics" sections → look for "Reduce JavaScript execution time" and "Minimize main-thread work". PSI will list specific scripts by name with their blocking time contribution
  • Step 2 — Chrome DevTools Coverage audit: DevTools → Coverage tab (Cmd+Shift+P → "Coverage") → record a page load → sort scripts by "Unused Bytes" descending. Scripts with 70%+ unused code are candidates for removal or code-splitting
  • Step 3 — Identify the biggest offenders: Common homepage culprits at this TBT level include: Intercom / live chat widgets (typically 400–600ms), Facebook Pixel + CAPI (200–400ms), HubSpot tracking (300ms+), A/B testing tools (Optimizely, VWO), and any un-deferred jQuery plugins
  • Step 4 — Defer or async: Add defer to all non-critical <script> tags. For GTM-managed scripts, change trigger from "Page View" to "Window Loaded"
  • Step 5 — Consider a script manager: If WP Rocket is installed, enable "Delay JavaScript Execution" globally with exceptions for critical scripts only
  • Target: Desktop TBT under 600ms in the first pass, under 300ms as the final goal
